MindSync

EEG · affect estimation · pipeline & evaluation

Overview

MindSync started with a pretty simple question: can you actually get anything meaningful about emotional state from EEG, especially in messy, real-world conditions? Not lab-perfect signals, but the kind you’d get in a student setup, short recordings, inconsistent contact, and labels that aren’t always reliable.

It quickly became clear this wasn’t going to be about finding the “best model.” It was more about building a pipeline that could handle everything from raw signals to something that could actually be evaluated and trusted.

What made it difficult

EEG is just messy. There’s noise everywhere, movement artifacts, drift, and inconsistencies between sessions and people. Early on, models seemed to perform well, but it didn’t feel real. Looking closer, the issue came from data leakage across sessions, meaning the model was effectively memorizing patterns tied to specific recordings.

On top of that, the labels themselves weren’t strong. Emotional state is subjective, and the dataset was imbalanced, so accuracy alone didn’t really mean anything.

Approach

Preprocessing was treated as part of the core system, not just a setup step. Filtering, windowing, normalization, and segmentation were all actively explored and iterated on.

The modeling pipeline was built in PyTorch, keeping things flexible to allow fast iteration and traceable debugging. The goal was to make sure that whenever performance changed, it could be clearly attributed to either the data, the model, or the evaluation setup.

Iteration and debugging

At first, the focus was on improving metrics, but that quickly broke down. The model was making confident but incorrect predictions on noisy segments, which forced a shift in how performance was being evaluated.

Error patterns were analyzed across sessions, and the data splitting strategy was tightened to avoid leakage. Some augmentations that initially seemed helpful actually degraded performance, especially given the existing label noise.

Evaluation

Instead of relying on a single aggregate metric, evaluation focused on where the model failed. Performance was broken down by session, class, and signal quality, which provided a much clearer understanding of model behavior.

This made it obvious that certain parts of the dataset were inherently harder, and treating all samples equally led to misleading conclusions.

Takeaways

The biggest takeaway from this project is that in this kind of machine learning setting, the pipeline matters more than the model. Architectures can be swapped easily, but if data handling and evaluation are not reliable, the results don’t carry much meaning.

This project fundamentally changed how machine learning systems are approached, especially when working with noisy, real-world data under tight constraints.
